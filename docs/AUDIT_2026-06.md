## Executive summary

**Build passes; lint fails on 5 trivial errors. The app's day-to-day flows
(signup, quote request, pro response, accept, bookings, reviews, portfolio,
contact) all work end-to-end. The money layer and the PII wall do not.**

Seven CRITICAL findings, in two clusters:

**1. Revenue is not trustworthy with Stripe LIVE.** Every subscription,
payment, dunning, and dispute DB write inside the Stripe webhook runs as the
`anon` role (cookie client in a sessionless context) with errors swallowed —
so $79/$199 purchases, cancellations, payment history, and chargebacks are
silently never persisted (BUG-02). Independently, the handlers read fields the
pinned Stripe "Basil" API removed, which crashes or no-ops them even before
RLS matters (BUG-01). The new $30 lead-unlock path got the client right but
has a 0-row phantom-success + session-overwrite race that can take a pro's
money without unlocking, or charge $60 for one lead (BUG-03).

**2. The PII wall — the product's core revenue gate — is bypassable for free
and undeliverable for pay.** Pros get the customer's full name + email from
the messaging API (SEC-01) and name + email + street address from the
auto-created booking the moment a quote is accepted (SEC-02) — no $30 paid.
Meanwhile a pro who DOES pay stays PII-masked forever, because nothing ever
writes `conversations.quote_request_id`, the flag the unlock gate checks
(FEAT-01). The well-built pieces (pro view, send-time filter, webhook capture)
are guarding doors while the walls are open.

Plus one live vulnerability: `claim_lead_with_credit`, an orphaned SECURITY
DEFINER function with no auth check and default PUBLIC EXECUTE — callable by
anyone holding the anon key (SCH-01).

13 HIGH findings follow (phantom writes against missing policies, two
unauthenticated admin-email endpoints, a booking-photos IDOR, auth-flow writes
that silently no-op, definer/RPC governance). Secrets hygiene is largely
clean: no live Stripe keys anywhere in tree or history; two SOTSVC anon-key
JWTs sit in old commits on a stale branch.

**Read this first:** Consolidated findings table below → then the
9-slice remediation order (smallest safe slices first; ⚠ = needs your
sign-off because it touches schema, RLS, or live Stripe).

## Section 1 — SECURITY

### 1.1 PII wall (pii-filter → quote_requests_pro_view → messaging unlock)

The purpose-built pieces are well-engineered — `quote_requests_pro_view`
correctly excludes address/customer_id/email/phone/TCPA and is what the pro
quote-browse uses (`app/dashboard/pro/quote-requests/actions.ts:99,113`); the
unlock gate requires `lead_acceptances.status='captured'` (a `pending` unpaid
row never unlocks); the webhook flips `pending → captured` only on a
signature-verified `checkout.session.completed` keyed by the server-set
`stripe_checkout_session_id`. But the wall leaks badly **around** those pieces:

**SEC-01 · CRITICAL — Messaging API hands pros the customer's full name + email with zero payment.**
`app/api/messages/route.ts:61` (conversation list) and
`app/api/messages/[conversationId]/route.ts:34` (single conversation) both
select `customer:users!conversations_customer_id_fkey(id, full_name, email)`
and return it raw to any conversation participant — including pros, pre-unlock.
No `lead_acceptances` check exists in either GET path. The UI only renders
`full_name`, but the API response carries the email. *Verified directly.*
Fix: drop `email` from both selects; return first-name-only for the pro side
(mirror the pro-view pattern) or gate full identity behind a captured
`lead_acceptances` row. App code only. **Human review: no.**

**SEC-02 · CRITICAL — Quote-accept auto-creates a booking that exposes name + email + street address, bypassing the $30 unlock entirely.**
`app/api/quotes/[id]/accept/route.ts:138-151` copies `address: quote.address`,
`zip_code`, and `special_instructions` into `bookings` at accept time.
`app/api/cleaner/bookings/route.ts:29-40` then serves the pro
`select('*', customer:users(id, full_name, email))` for their bookings, and
bookings RLS allows pro SELECT (`supabase/11-bookings-schema.sql:50-56`).
Since `quote.status==='accepted'` is exactly the unlock precondition, the pro
gets the full PII set free at the same moment the paid unlock becomes
available. *Verified directly.* Fix: don't copy address/instructions into the
booking (or withhold them + the users join from `cleaner/bookings`) until a
captured `lead_acceptances` row exists. App code; revenue path — **human review: yes**.

**SEC-03 · HIGH — PII filter is API-only; RLS permits direct client INSERT into `messages`.**
`filterPII`/`filterPIIWithWindow` run only inside POST `/api/messages`
(`app/api/messages/route.ts:271-344`). The messages INSERT policies
(`supabase/13-messaging-schema.sql:96-116`) let authenticated customers/pros
insert directly, and the browser holds a live Supabase client — a console
`supabase.from('messages').insert(...)` carries raw phone/email straight past
the filter, and the realtime subscription (`lib/hooks/useMessages.ts:59-79`)
delivers it. No DB trigger enforces filtering (the DLD-224 migration adds only
the log table). Fix: BEFORE INSERT trigger enforcing the filter, or revoke
direct INSERT and force writes through the API. **Touches RLS/schema — human review: yes.**

**SEC-04 · MEDIUM — Pro notifications embed customer full name pre-payment.**
`app/api/messages/route.ts:148` sets `senderName = userData.full_name`; the
new-message email (`lib/email/new-message.ts:31,67` — subject "New message
from {full_name}") and SMS (`app/api/messages/route.ts:388-391`) deliver it to
the pro with no unlock check. Fix: first-name-only until unlock. App code only.

**SEC-05 · MEDIUM — `lib/pii-filter.ts` bypass vectors (where it does run).**
`lib/pii-filter.ts:23-61,145-209`: ASCII-only `\d`/`[a-zA-Z]` patterns miss
full-width/Arabic-Indic digits and homoglyphs; `name [at] gmail [dot] com`
evades the email regex and the spelled-out map; base64-encoded contact info
passes; the rolling window reads only the last 5 messages
(`lib/pii-filter.ts:310`), so a phone split across 6+ messages evades the
cumulative count; `special_instructions` and upload captions are unfiltered.
Fix: NFKC-normalize + confusables map, add `[at]`/`[dot]`, widen the window.
App code only. Note SEC-03 makes these moot for a determined attacker until
the DB-level enforcement lands.

### 1.3 Phantom writes (code writes vs live RLS policies)

**SEC-15 · CRITICAL — Stripe webhook subscription handlers use the cookie/anon client; subscription state writes are RLS-governed with no session.**
`app/api/webhooks/stripe/route.ts:110-147` routes subscription/invoice events
into `lib/stripe/subscription-service.ts`, which builds the cookie-bound
server client (`subscription-service.ts:2,10-12` — *verified directly*) and
writes `subscriptions` (insert `:126`, updates `:139,:167,:198`) and `pros`.
In webhook context there is no user session, so either RLS silently blocks
these writes (paying subscribers whose tier/status never persists — a live
revenue bug) or anon writes to `subscriptions`/`pros` are allowed (an RLS
hole). The lead-unlock branch and the accept route were both deliberately
moved to service-role for exactly this reason; the subscription path wasn't.
`lib/stripe/webhook-event-service.ts:13` has the same anon-client assumption —
safe only because its RPCs should be SECURITY DEFINER (verify).
Fix: `createServiceRoleClient()` in subscription/invoice/dunning handlers.
**Touches Stripe + RLS — human review: yes. Verify live `subscriptions`/`pros`
policies to classify which failure mode is live.**

**SEC-16 · CRITICAL — Chargeback disputes are silently never recorded.**
`lib/stripe/disputes.ts:8,20` uses the cookie client inside webhook handlers
(`app/api/webhooks/stripe/route.ts:161-167`); live facts: `disputes` has ZERO
INSERT and ZERO UPDATE policies, so the insert at `disputes.ts:130` and
updates at `:185-208` are RLS-blocked, and the insert error is never checked.
With Stripe LIVE, a real chargeback would flow through email side-effects but
leave no DB record and no cleaner flag. *Client import verified directly.*
Fix: service-role client here (same as lead-unlock branch). **Human review:
yes (Stripe path).**

**SEC-17 · HIGH — Admin approve/reject in-app notifications silently fail (notifications INSERT is service-role-only).**
`app/dashboard/admin/actions.ts:52-70` (`createNotification`) inserts into
`notifications` with the admin's user client; the live policy allows INSERT
only to service-role, so every pro-approval/rejection in-app notification is
dropped (error only logged). Fix: switch to `createServiceRoleClient()`.
**Human review: no.**

**SEC-18 · HIGH — `cleaner_documents` DELETE endpoint is a guaranteed phantom that reports success.**
`app/api/cleaners/documents/route.ts:140-144` deletes with the pro's user
client; live `cleaner_documents` has NO DELETE policy, so 0 rows match, no
error returns, and the route replies `{success:true}` while the document (and
its storage object) survives. Repo SQL *does* define the policy
(`supabase/06-onboarding-wizard.sql:67`) — this is repo-vs-live policy drift.
No UI currently calls this endpoint (it's also dead code, QUAL-03), which
contains the blast radius. Fix: delete the route, or re-apply the DELETE
policy as a timestamped migration / service-role delete after the ownership
check, with a row-count assertion. **Touches RLS — human review: yes.**

**SEC-19 · HIGH — OAuth email-based account linking silently does nothing.**
`app/auth/callback/route.ts:135` re-points an existing `users` row to a new
auth id via `update({id: user.id}).eq('email', ...)` with the user client —
under a self-row UPDATE policy the old-id row can never match → 0 rows,
unchecked, and the user is redirected as if linked; their old data stays
orphaned. (Lines 108, 146, 163 in the same file are also fire-and-forget.)
Fix: do the link with the service-role client (email ownership already proven
by OAuth) and check errors. **Human review: yes (auth).**

**SEC-20 · HIGH — Post-signup profile writes run as `anon` when email confirmation is pending.**
`components/auth/AuthForm.tsx:155-162` (users), `:178-183` (pros),
`:194-234` (service_areas) execute right after `signUp()`; with email
confirmation on, `session` is null → all writes run as anon, all RLS-blocked,
none check `error`. Full name, phone, business name, and the seed service
area silently never save for confirm-required signups. Fix: move enrichment
server-side (the metadata is already passed at `:135-139` — extend the
`handle_new_user` trigger, or a service-role server action). **Human review:
yes if trigger (schema).**

**SEC-21 · MEDIUM — `users.role` may be writable from the browser.**
`app/auth/select-role/page.tsx:62-68` updates `role` directly from the client
and it evidently works (the flow depends on it). If the live `users` UPDATE
policy doesn't column-restrict `role`, any authenticated user can
`update({role:'admin'})` from the console — and every admin gate trusts that
column. **Verify the live policy first; if unrestricted this is CRITICAL.**
Fix: trigger or column-level check constraint on role transitions.
**Touches RLS/schema — human review: yes.**

**SEC-22 · MEDIUM — Webhook/billing RPCs are SECURITY DEFINER with default PUBLIC EXECUTE.**
`supabase/08-stripe-webhooks.sql:90-219` — `record_webhook_event`,
`mark_webhook_processed/failed`, `is_webhook_processed`,
`update_cleaner_billing_dates`, `increment_payment_failed_count` grant to

**SEC-23 · MEDIUM — Cross-party `messages` updates may be phantom (reports + read receipts).**
`app/api/messages/report/route.ts:86-93` updates the *other* participant's
message; `app/api/messages/[conversationId]/route.ts:152-157` stamps
`read_at` on the other side's rows. If the live messages UPDATE policy is
sender-scoped, both match 0 rows and return success (0 rows ≠ error).
Symptoms: reports never reach admins; unread badges never clear. Verify
policy; if sender-scoped, route through service-role after the existing

## Section 2 — BUGS (logic errors, races, phantom successes, status contracts)

**BUG-01 · CRITICAL — Stripe "Basil" SDK: subscription webhook handlers read fields that no longer exist on the event objects.**
`lib/stripe/config.ts:16` pins `apiVersion: '2025-07-30.basil'` (stripe SDK
`^18.4.0`). Under Basil, `Subscription.current_period_start/end` moved to
subscription *items*, and `Invoice.subscription`/`Invoice.payment_intent`
moved to `invoice.parent.subscription_details`/`invoice.payments` — confirmed
against the installed SDK types (`node_modules/stripe/types/Subscriptions.d.ts`
has zero occurrences of `current_period_end`). Consequences:
- `handleSubscriptionCreated/Updated` (`lib/stripe/subscription-service.ts:106-121,158-162`):
  `new Date(undefined * 1000)` → RangeError → all 3 retries fail → webhook 500s
  forever → **a $79/$199 purchase is never recorded; the pro never upgrades.**
- `handlePaymentSucceeded/Failed` (`:223-234,:285-296`): `inv.subscription` is
  undefined → silent early return → no payments rows, no dunning, no monthly
  lead-count reset — and the event is marked processed.
Caveat: the event shape follows the *webhook endpoint's* pinned version in the
Stripe dashboard — verify it; if pinned pre-Basil the dates survive, but
BUG-02 still kills every write. Fix: read period from
`subscription.items.data[0].current_period_*`; resolve invoice→subscription
via `invoice.parent?.subscription_details`. **Stripe — human review: yes.**

**BUG-02 · CRITICAL — (= SEC-15/SEC-16) Webhook persistence runs as `anon` and swallows every error.**
Full mechanics in §1.3: `lib/stripe/subscription-service.ts` (10 write sites),
`lib/stripe/dunning.ts` (6), `lib/stripe/disputes.ts` (4) all use the
cookie client in a sessionless webhook context; repo RLS
(`supabase/02-row-level-security.sql:142-164`) gives `subscriptions`/`payments`
SELECT-only policies (the comment at `:150` literally says "service key
bypasses RLS" — but the code never uses it). Not one write checks `error`, so
events are marked processed and Stripe never retries. BUG-01 and BUG-02
compound: fixing one without the other leaves subscription revenue
unrecorded. Fix: `createServiceRoleClient()` + error/row-count checks in all
three services. **Stripe + RLS — human review: yes.**

**BUG-03 · CRITICAL — Lead unlock: 0-row webhook UPDATE phantom success + session-overwrite race → paid-but-locked lead, possible $60 double charge.**
(a) The webhook capture (`app/api/webhooks/stripe/route.ts:86-98`) updates by
`.eq('stripe_checkout_session_id', session.id)` and checks only `error` — a
0-row match is silent, and the event is marked processed permanently.
(b) How 0 rows happens: the unlock route's resume guard
(`app/api/leads/[quoteId]/unlock/route.ts:74-81`) only resumes when the prior
session is `'open'`. If the pro pays and re-clicks before the webhook lands
(row still `pending`, session `complete`), the code mints a fresh session and
**overwrites `stripe_checkout_session_id`** (`:156-160`). The delayed webhook
for the *paid* session now matches 0 rows → stuck `pending`, money taken —
and the second open session can be paid too ($60 for one $30 unlock). The
`UNIQUE(quote_request_id, cleaner_id)` constraint prevents duplicate rows,
not duplicate sessions. Fix: webhook updates by `metadata.lead_acceptance_id`
(already in metadata, `unlock/route.ts:134`) with `.select('id')` and throw on
0 rows so Stripe retries; unlock route must not mint a new session when the
prior one is `complete`/`payment_status==='paid'`, and should
`stripe.checkout.sessions.expire` the old one before creating a replacement.
**Stripe — human review: yes.**

**BUG-04 · HIGH — Webhook events stuck in `processing` are ACKed with 200 → Stripe stops retrying → event lost forever.**
`app/api/webhooks/stripe/route.ts:232-235` + `supabase/08-stripe-webhooks.sql:104-140`:
if the function dies between `record_webhook_event` and mark-processed/failed
(deploy, crash, or serverless timeout — the in-handler retry loop sleeps
1s+2s+4s at `route.ts:13-14,44-46`, easily breaching a 10s Netlify limit),
the row stays `processing` and every Stripe retry gets
`200 {processing:true}`. Fix: treat `processing` rows older than a few
minutes as retryable, or return 500 for `processing`. **RPC/schema — human
review: yes.**

**BUG-05 · HIGH — `respondToQuote` claim has a TOCTOU race and 0-row phantom success.**
`app/dashboard/pro/quote-requests/actions.ts:178-212`: availability is
checked by read, then the claim UPDATE filters only `.eq('id', quoteId)` —
no `.is('cleaner_id', null)`, no `.eq('status','pending')`, no row-count
check. Two pros responding concurrently: the second silently overwrites the
first's claim (and the customer is emailed both quotes). This is exactly the
bug class PR #60 fixed in the accept route, unfixed here. Fix: add the guard
